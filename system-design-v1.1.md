# 1. システム概要

## 1.1 目的

本システムは、学校の授業予定組立から学生グループのミーティング予約までを一貫して管理する統合スケジューリングプラットフォームです。以下の4つのモジュールで構成されます。

## 1.2 モジュール構成

| **Module** | **名称**                | **概要**                                               | **アクセス範囲**          |
|------------|-------------------------|--------------------------------------------------------|---------------------------|
| M1         | 授業予定組立ツール      | 講師・カリキュラム・教室を入力し、DPで時間割を自動生成 | 教務管理者のみ            |
| M2         | データ統合              | 授業予定・Googleカレンダー・個人予定を統合             | システム内部              |
| M3         | オートスケジューラ      | グループの空きコマ・空き教室を自動計算                 | グループメンバー          |
| M4         | 予約システム            | 空きコマに予定を登録し共有                             | 全ユーザー（公開情報）    |
| M5         | Webhook・リマインド通知 | 予定変更Webhook・リマインド・通知配信                  | 全ユーザー + 外部サービス |

## 1.3 データフロー概観

M1（授業予定組立）→ M2（データ統合）→ M3（空き計算）→ M4（予約登録）→ M5（通知配信）

各モジュールのデータは一方向に流れ、M1の出力がM2の入力となり、M4の予約結果はM2へフィードバックされます。M5はM1～M4全モジュールのイベントを購読し、Webhook配信・リマインド通知を横断的に処理します。

# 2. Module 1: 授業予定組立ツール

## 2.1 機能要件

- 講師データ（名前・専攻・担当科目・空き時間）のCSVインポート
- カリキュラムデータ（科目名・専攻・週当たりコマ数・必要教室タイプ）の管理
- 教室データ（室名・定員・タイプ）の管理
- 動的計画法による自動時間割生成（詰め込みモード / 分散モード）
- コマ入れ替え機能（候補ハイライト付き）
- カリキュラム設定の期間制限（一定期間内のみ編集可能）

## 2.2 スケジューリングアルゴリズム

### 2.2.1 動的計画法（DP）による配置

制約充足問題（CSP）としてモデル化し、以下の手順で配置します。

1. 各カリキュラムの候補スロット数を計算
2. 候補が少ない順（MRV: Minimum Remaining Values）にソート

3. 各カリキュラムを順番に配置、制約伝播で矛盾を検出
4. バックトラックで配置不可時に別候補を試行

### 2.2.2 モード切替

| **モード**      | **戦略**                                           | **ユースケース**             |
|-----------------|----------------------------------------------------|------------------------------|
| 詰め込み (Pack) | 制約の厳しい科目から優先配置。早い曜日・コマに密集 | 教室稼働率最大化、省エネ     |
| 分散 (Spread)   | 曜日をまたいで均等配置。ラウンドロビン方式         | 学生の負荷分散、空きコマ確保 |

### 2.2.3 入れ替え候補ハイライト

配置済みの授業コマをクリックすると、入れ替え可能な先がハイライト表示されます。

**入れ替え対象の条件:**

- 入れ替え先候補のコマが空欄または入れ替え可能な授業であること
- 対象コマに対応可能な教室が存在すること
- 講師の空き時間と矛盾しないこと

**候補数による色分け:**

| **候補数** | **色**             | **意味**                    |
|------------|--------------------|-----------------------------|
| ≤ 3コマ    | 灰色 (#6E7681)     | 候補少 — 入れ替え困難       |
| 4～6コマ   | オレンジ (#D29922) | 候補中 — 慎重に選択         |
| ≥ 7コマ    | オレンジ (#D29922) | 候補中 — 柔軟性あり         |
| ≥ 15コマ   | 緑 (#3FB950)       | 候補多 — 再計算スキップ対象 |

### 2.2.4 無限ループ防止

入れ替え操作時の再帰的再計算による無限ループを防止するため、以下のルールを適用します。

- 配置時の候補が15コマ以上の講師の授業は再計算対象外とする
- 入れ替えの連鎖は最大2段階までとする（直接入れ替え + 影響先の再配置）
- 各入れ替え操作にTransaction IDを付与し、同一Transaction内の再入を禁止

## 3.1 機能要件

- M1の確定授業予定データの取得
- Googleカレンダーからの学校行事予定の取得
- 各メンバーの個人カレンダーデータの取得（M3用）
- データの正規化・統合・キャッシュ

## 3.2 データソース

| **ソース**       | **データ内容**       | **取得方法**                | **更新頻度**              |
|------------------|----------------------|-----------------------------|---------------------------|

## 3.3 プライバシールール

本モジュールのデータは機密情報として扱い、以下のルールを適用します。

| **データ種別** | **公開範囲**       | **他者への公開内容**                    |

## 3.5 キャッシュ戦略

- M1データ: 学期単位でキャッシュ、変更時に無効化

- 個人カレンダー: ユーザーリクエスト時に取得（10分TTL）
- M4フィードバック: イベントドリブンで即時反映

# 4. Module 3: オートスケジューラ

## 4.1 機能要件

- グループの作成・編集・削除
- グループメンバーの空きコマ自動計算
- 空き教室の自動検索
- 登校日・共通空き曜日の可視化
- MTG候補のランキング提案

## 4.2 グループモデル

| **フィールド** | **型**        | **説明**               |
|----------------|---------------|------------------------|
| id             | string (UUID) | グループID             |
| name           | string        | グループ名             |
| members        | string\[\]    | メンバーのuserIDリスト |
| createdBy      | string        | 作成者                 |
| createdAt      | Date          | 作成日時               |

## 4.3 空き計算アルゴリズム

以下の手順でグループ全員の空きコマを計算します。

1. M2から各メンバーの UnifiedSlot マトリクスを取得
2. 各コマについて、全メンバーの status を確認

3. 全員 status=free のコマを「全員空き」として抽出
4. 70%以上が free のコマを「部分空き」として抽出

5. 空き教室を M2 の教室使用状況から照合

## 4.4 ランキングスコア

MTG候補コマに以下のスコアを付与し、降順で提案します。

| **要素**       | **スコア** | **説明**                     |
|----------------|------------|------------------------------|
| 参加可能人数   | +10/人     | 全員参加可能なら最高スコア   |
| 全員登校日     | +20        | 全員が登校している曜日のコマ |
| 空き教室あり   | +15        | 対応する空き教室がある       |
| 連続コマ       | +5         | 前後のコマも空き（延長可能） |
| 土日ペナルティ | -5         | 土日のコマは優先度下げ       |

## 4.5 UI設計

- メンバー選択パネル: カードUIでメンバーをチェックボックス選択
- 空きコマグリッド: 7曜日×11コマのヒートマップ表示
- 候補提案リスト: スコア降順でカード表示、クリックでM4予約へ遷移
- 登校日サマリ: 各メンバーの登校曜日をタグ表示
- 授業名非公開: グリッド上は「予定あり」のみ表示、専攻名も非公開

# 5. Module 4: 予約システム

## 5.1 機能要件

- M3の候補コマから予約を登録
- 空き教室の予約・確保
- 予約内容の他者への共有（公開情報）
- 予約のキャンセル・変更
- 予約データのM2へのフィードバック

## 5.2 データモデル

### 5.2.1 Reservation（予約）

| **フィールド** | **型**        | **説明**                        |
|----------------|---------------|---------------------------------|
| id             | string (UUID) | 予約ID                          |
| groupId        | string        | グループID                      |
| title          | string        | 予約タイトル（公開）            |
| day            | number (0-6)  | 曜日                            |
| period         | number (0-10) | コマ                            |
| roomId         | string        | 教室ID                          |
| createdBy      | string        | 予約者                          |
| participants   | string\[\]    | 参加者IDリスト                  |
| status         | enum          | confirmed / cancelled / pending |
| createdAt      | Date          | 作成日時                        |
| note           | string        | メモ（公開）                    |

## 5.3 公開ルール

予約情報は公開情報として扱い、以下が全ユーザーに公開されます。

| **項目**             | **公開** | **非公開** |
|----------------------|----------|------------|
| 予約タイトル         | ○        |            |
| 曜日・コマ           | ○        |            |
| 教室                 | ○        |            |
| グループ名           | ○        |            |
| 参加者名             | ○        |            |
| 予約者               | ○        |            |
| メモ                 | ○        |            |
| 各メンバーの授業情報 |          | ○          |
| 各メンバーの個人予定 |          | ○          |

## 5.4 予約フロー

1. M3でMTG候補コマを確認
2. 「予約へ」ボタンでM4に遷移

3. タイトル・メモを入力
4. 教室を選択（M3から自動提案済み）

5. 確定ボタンで予約登録
6. 全ユーザーに公開・通知

7. 予約データをM2へフィードバック（教室・コマの使用状況更新）

## 5.5 競合制御

同一教室・コマへの同時予約を防止するため、以下の制御を行います。

- 楽観的ロック: 予約確定時にバージョンチェック、競合時は後発を拒否
- タイムアウト: 予約確定画面では5分以内に操作を完了する必要あり
- 予約キュー: 同時リクエスト時はFIFOで処理

# 6. Module 5: Webhook・リマインド通知

## 6.1 機能要件

- 予定変更時のWebhook配信（授業変更・予約作成・キャンセル）
- MTG開始前のリマインド通知（設定可能なタイミング）
- 通知チャンネルの選択（アプリ内・メール・外部Webhook）
- ユーザーごとの通知設定（オプトイン・オプトアウト）
- Webhookエンドポイントの登録・管理（外部サービス連携用）
- 配信履歴・リトライログの管理

## 6.2 イベントソース

他モジュールで発生するイベントを購読し、通知処理をトリガーします。

| **ソース** | **イベント名**        | **トリガー条件**   | **通知内容**                       |
|------------|-----------------------|--------------------|------------------------------------|
| M1         | schedule.confirmed    | 授業予定確定時     | 新学期時間割確定のお知らせ         |
| M1         | schedule.changed      | 授業コマ変更時     | 授業変更通知（影響専攻の学生向け） |
| M4         | reservation.created   | 予約作成時         | 予約確定通知（参加者全員）         |
| M4         | reservation.updated   | 予約変更時         | 変更内容通知（日時・教室・参加者） |
| M4         | reservation.cancelled | 予約キャンセル時   | キャンセル通知（参加者全員）       |
| M4         | reservation.reminder  | 予約開始前         | リマインド通知（設定時刻）         |
| M2         | sync.conflict         | カレンダー競合検出 | 予定競合警告（本人のみ）           |

## 6.3 通知チャンネル

| **チャンネル** | **配信方法**    | **対象**               | **用途**                           |
|----------------|-----------------|------------------------|------------------------------------|
| アプリ内通知   | WebSocket / SSE | 全ユーザー             | リアルタイムの予定変更・リマインド |
| メール通知     | SMTP / SendGrid | オプトインユーザー     | 予約確定・変更・リマインド         |
| 外部Webhook    | HTTP POST       | 登録済みエンドポイント | Slack・ Discord・外部システム連携  |
| プッシュ通知   | FCM / APNs      | モバイルユーザー       | リマインド・緊急変更               |

## 6.4 リマインド設計

### 6.4.1 リマインドタイミング

ユーザーが予約ごとにリマインドタイミングを設定できます。デフォルト値はユーザー設定で変更可能です。

| **タイミング**     | **デフォルト** | **設定範囲**      | **対象チャンネル**         |
|--------------------|----------------|-------------------|----------------------------|
| 前日リマインド     | ON             | 前日 18:00～21:00 | メール・プッシュ・アプリ内 |
| 当日朝リマインド   | ON             | 当日 07:00～09:00 | プッシュ・アプリ内         |
| 直前リマインド     | ON             | 15分前～60分前    | プッシュ・アプリ内         |
| カスタムリマインド | OFF            | 任意の時刻指定    | 全チャンネル               |

### 6.4.2 リマインドスケジューラ

リマインドはジョブスケジューラ（cronベース）で管理します。

1. 予約作成時にリマインドジョブをスケジューラに登録
2. スケジューラが1分間隔で配信対象をポーリング

3. 配信時刻に達したジョブをキューに投入
4. ワーカーがチャンネル別に配信処理を実行

5. 配信結果をログに記録（成功・失敗・リトライ）

予約がキャンセルされた場合、未配信のリマインドジョブは自動的に削除されます。

## 6.5 Webhook設計

### 6.5.1 Webhookエンドポイント登録

外部サービスがイベントを受信するためのWebhook URLを登録できます。

| **フィールド**  | **型**        | **説明**                      |
|-----------------|---------------|-------------------------------|
| id              | string (UUID) | Webhook登録ID                 |
| url             | string (URL)  | 配信先URL                     |
| events          | string\[\]    | 購読イベント名のリスト        |
| secret          | string        | HMAC-SHA256署名用シークレット |
| isActive        | boolean       | 有効/無効                     |
| createdBy       | string        | 登録者                        |
| failCount       | number        | 連続失敗カウント              |
| lastDeliveredAt | Date \| null  | 最終配信日時                  |

### 6.5.2 Webhookペイロード形式

全イベント共通のJSONペイロード形式です。

| **フィールド** | **型**            | **説明**                              |
|----------------|-------------------|---------------------------------------|
| event          | string            | イベント名 (e.g. reservation.created) |

### 6.5.3 セキュリティ

- HMAC-SHA256署名: 各配信のX-Signatureヘッダーに署名を付与
- タイムスタンプ検証: 5分以上古いペイロードは拒否（リプレイ攻撃防止）
- シークレットローテーション: 管理画面からシークレットを再生成可能
- IPホワイトリスト: オプションで配信元IPを制限可能

## 6.6 配信信頼性

### 6.6.1 リトライ戦略

Webhook配信が失敗した場合、指数バックオフでリトライします。

| **リトライ回数** | **待機時間** | **最大待機** | **アクション**                 |

## 6.8 アーキテクチャ概要

通知システムはイベントドリブンアーキテクチャを採用します。

1. 各モジュールがイベントバス（Redis Pub/Sub）にイベントを発行
2. M5のイベントハンドラが購読・受信

3. ルーティングエンジンがイベント種別×ユーザー設定を照合し、配信先を決定
4. チャンネル別ワーカーが非同期で配信実行

5. 配信結果をログに記録、失敗時はリトライキューに再投入

## 8.2 リスクと対策

| **リスク**                          | **影響度** | **対策**                                     |
|-------------------------------------|------------|----------------------------------------------|
| Google Calendar APIのレートリミット | 中         | キャッシュ戦略 + バッチ取得                  |
| DPスケジューラの計算時間増大        | 高         | MRVヒューリスティク + タイムアウト制限       |
| 予約競合                            | 中         | 楽観的ロック + FIFOキュー                    |
| 個人情報漏洩                        | 高         | プライバシールールの厳格運用 + 監査ログ      |
| 入れ替え無限ループ                  | 高         | 候補15+スキップ + 2段階制限 + TxID           |
| Webhook配信先障害                   | 中         | 指数バックオフリトライ + 5回失敗で自動無効化 |
| リマインドの大量配信                | 中         | BullMQキューによるレート制御 + バッチ処理    |
| 通知スパムリスク                    | 低         | Quiet Hours設定 + ユーザー単位のレート制限   |

# 9. 付録

## 9.1 用語定義

| **用語**     | **説明**                                                                 |
|--------------|--------------------------------------------------------------------------|
| コマ         | 1時間単位の授業スロット（9:30開始、1～11限）                             |
| DP           | 動的計画法 (Dynamic Programming)                                         |
| CSP          | 制約充足問題 (Constraint Satisfaction Problem)                           |
| MRV          | Minimum Remaining Values（最小残余値ヒューリスティク）                   |
| 統合スロット | M2が生成する、各データソースを統合したコマ単位の情報                     |
| TxID         | トランザクションID（入れ替え操作の追跡用）                               |
| FIFO         | First In First Out（先着順処理）                                         |
| Webhook      | HTTP POSTでイベントを外部サービスに配信する仕組み                        |
| HMAC-SHA256  | ハッシュベースメッセージ認証コード（Webhook署名検証用）                  |
| SSE          | Server-Sent Events（サーバーからクライアントへの一方向リアルタイム通信） |
| BullMQ       | Redisベースのジョブキューライブラリ（リマインド・リトライ用）            |
| Quiet Hours  | 通知を停止する時間帯設定（深夜帯など）                                   |
| deliveryId   | Webhook配信の寣等性保証用一意識別子                                      |
