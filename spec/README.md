# Schedula 仕様書

> 統合スケジューリングプラットフォーム Schedula の詳細仕様

## ディレクトリ構成

```
spec/
├── design/          # 各モジュールの設計仕様（ユースケース含む）
├── code/            # 各モジュールの実装仕様（依存関係・スキーマ・変数名）
├── test/            # 各モジュールのテスト要件
├── data/            # DBデータスキーマ（バックリンク付き）
└── ux/              # UXデザイン資料
```

## モジュール一覧

| Module | 名称 | 概要 |
|--------|------|------|
| Auth | 認証・認可 | JWT + Google OAuth によるユーザー認証 |
| Group | グループ管理 | グループの作成・メンバー管理 |
| Calendar | カレンダー | 個人予定・Google Calendar 連携 |
| MyPlan | マイプラン | 週間ルーティーン管理 |
| M1 | 授業予定組立ツール | カリキュラム管理・自動時間割生成・施設予約 |
| Smart Scheduler | オートスケジューラ | グループの空きコマ自動計算・候補提案 |
| Notification | 通知システム | Webhook・リマインド・マルチチャンネル通知 |
| Voting | 日程調整 | 投票ベースの日程調整 |
| Holiday | 休日管理 | 祝日・休業期間・グループイベント |
| Integrations | 外部連携 | Google Calendar・Notion 双方向同期 |
| Settings | システム設定 | アプリ全体の設定管理 |

## データフロー

```
M1（授業予定組立）→ M2（データ統合）→ M3（空き計算）→ M4（予約登録）→ M5（通知配信）
```

## 時間割定義

- 曜日: 月〜日（7日間、0=月〜6=日）
- コマ: 1限〜11限（9:30開始、各1時間）
- 1コマ = 1時間、9:30スタート

## 凡例

- **design**: 何を作るか・なぜ作るか（ユースケース・機能要件・振る舞い）
- **code**: どう作るか（依存モジュール・データスキーマ・変数名・APIパス）
- **test**: どう検証するか（テストケース・境界条件・期待結果）
- **data**: データの定義（テーブル定義・制約・使用モジュールのバックリンク）
- **ux**: どう見せるか（画面構成・インタラクション・デザイン原則）
